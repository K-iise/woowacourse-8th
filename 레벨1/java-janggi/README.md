# java-janggi

장기 미션 저장소

## 기능 요구 사항
- 애플리케이션 실행 시 사용자는 다음 두가지 기능을 선택한.
  - 게임 불러오기
  - 새 게임 시작
- 게임 불러오기 선택 시 아래와 같은 진행이 시작된다.
  - 저장된 게임 목록을 출력한다.
  - 저장된 게임이 없는 경우 새 게임 시작으로 진행된다.
  - 사용자로부터 불러올 게임의 번호를 입력 받는다.
  - 입력값의 유효성을 검증한다.
  - 선택한 저장 데이터를 불러오고 게임을 시작한다.
- 새 게임 시작 시 보드 초기화를 진행한다.
  - 초나라(플레이어1) 플레이어는 번호로 제시된 마와 상의 4가지 유형(ex, 원앙상, 귀마, 양귀마, 양귀상)의 선택지를 출력한다.
  - 초나라 사용자로부터 제시된 선택지 번호를 입력 받는다.
  - 한나라(플레이어2) 플레이어는 번호로 제시된 마와 상의 4가지 유형(ex, 원앙상, 귀마, 양귀마, 양귀상)의 선택지를 출력한다.
  - 한나라 사용자로부터 제시된 선택지 번호를 입력 받는다.
- 게임이 시작되면, 플레이어는 턴을 번갈아 진행한다.(턴의 내용은 아래와 같다)
  - 매 턴 시작시 보드 상태를 갱신하여 출력한다.
  - 플레이어로부터 이동할 말의 이름을 입력받는다.
    - 입력받은 말의 이름이 유효한지 검증한다.
    - 입력받은 말이 살아있는지 검증한다.
  - 플레이어로부터 이동할 위치의 좌표를 입력받는다.
    - 입력받은 좌표가 장기 판의 크기를 벗어나지 않는지 검증한다.
    - 입력받은 좌표가 이동하려는 말의 이동 규칙에 맞는지 검증한다.
  - 입력 받은 위치로 기물을 이동 시킨다.
    - 이동한 위치에 상대방의 기물이 있다면, 상대방의 기물을 잡는다.
- 궁이 잡히면 게임이 종료된다.
  - 궁을 잡은 플레이어의 승리 메시지를 출력한다.(ex,"초나라 승리!", "한나라 승리!")

### 기물 이동 규칙 요구사항
**졸(Soldier)의 이동 방법**
1. 이동 방향 및 거리
   - [x] 전진 또는 좌/우 가로 방향으로 1칸만 이동 가능하다.
   - [x] 후진은 불가능하다.
2. 진영 별 전진 방향
   - [x] 초나라는 y의 좌표가 감소하는 방향이 전진이다.
   - [x] 한나라는 y의 좌표가 증가하는 방향이 전진이다.
3. 이동 제한
   - [x] 목적지에 아군 기물이 있는 경우 이동할 수 없다.
   - [x] 궁성 내부에서도 대각선 방향으로 이동할 때는 궁성 중심을 지나는 대각선만 가능하다.

**궁과 사(King)의 이동 방법**

1. 이동 방향 및 거리
   - [x] 가로, 세로, 대각선 방향으로 1칸만 이동 가능하다.
2. 이동 제한
   - [x] 목적지에 아군 기물이 있는 경우 이동할 수 없다.
   - [x] 궁성 밖으로 나갈 수 없으며, 궁성 내부에서도 대각선 방향으로 이동할 때는 궁성 중심을 지나는 대각선만 가능하다.

**마(Horse)의 이동 방법**

1. 이동 방향 및 거리
   - [x] 1단계 : 상,하,좌,우 1칸 이동한다.
   - [x] 2단계 : 1단계 이동 방향을 기준으로 대각선 앞 방향으로 1칸 이동한다.
2. 이동 제한
   - [x] 1단계에서 기물이 있는 경우 이동할 수 없다.
   - [x] 2단계에서 아군 기물이 있는 경우 이동할 수 없다.
   - [x] 2단계에서 상대 기물이 있는 경우 잡으면서 이동한다.

**상(Elephant)의 이동 방법**

1. 이동 방향 및 거리
    - [x] 1단계 : 상,하,좌,우 한 칸 이동한다.
    - [x] 2단계 : 1단계 이동 방향을 기준으로 대각선 방향으로 두 칸 이동한다.
2. 이동 제한
    - [x] 1단계 칸 또는 2단계 경로 중간에 기물이 있는 경우 이동할 수 없다.
    - [x] 목적지에 아군 기물이 있는 경우 이동할 수 없다.
    - [x] 목적지에 상대 기물이 있는 경우 잡으면서 이동한다.

**차(Chariot)의 이동 방법**

1. 이동 방향 및 거리 
   - [x] 상,하,좌,우 이동이 가능하다.
2. 이동 제한
   - [x] 이동하려는 경로에 기물이 없어야 한다.
   - [x] 궁성 내부에서 대각선 방향으로 이동할 때는 궁성 중심을 지나는 대각선만 가능하다.

**포(Cannon)의 이동 방법**

1. 이동 방향 및 거리
    - [x] 상,하,좌,우 직선 방향으로만 이동한다.
2. 이동 제한
    - [x] 출발지와 목적지 사이에 **포가 아닌 기물이 정확히 하나** 있어야 한다. (포는 포를 넘지 못한다.)
    - [x] 위 조건을 만족할 때만 빈 칸으로 이동하거나 상대 기물을 잡을 수 있다.
    - [x] 목적지에 아군 기물이 있는 경우 이동할 수 없다.
    - [x] 궁성 내부에서 대각선 방향으로 이동할 때는 궁성 중심을 지나는 대각선만 가능하다.

## 토론 규칙

### 1. 테스트 단위 및 제외 기준
- **(If-Then)** 만약 기능이 `public` 메서드로 외부에 노출되어 있다면 → 그 기능 단위로 테스트를 작성한다 (메서드 하나가 아니라, 하나의 기능을 구성하는 흐름 단위).
- **(If-Then)** 만약 `private` 메서드를 직접 테스트하고 싶어진다면 → 클래스 분리 또는 접근제한자 재설계가 필요하다는 신호로 간주하고, 리팩터링을 먼저 검토한다.
- **(If-Then)** 만약 경계값이 존재하는 로직이라면 → 경계값 기준으로 테스트 케이스를 나눈다.
- **(If-Then)** 만약 유틸/파싱 메서드에 비즈니스 규칙(도메인 규칙 + 형식 규칙)이 포함되어 있지 않다면 → 테스트에서 제외한다 (도메인과 무관한 단순 변환은 제외).
- **(금지)** 이번 미션에서 단순 `getter/setter`, `DTO`는 테스트하지 않는다.
- **(금지)** 이번 미션에서 입출력(콘솔 I/O, 컨트롤러) 자체는 테스트하지 않는다.

### 2. 테스트 어려움 대응 및 리팩터링 우선순위
- **(If-Then)** 만약 테스트 작성이 어렵다면 → 테스트 자체를 포기하지 않고, 설계에 문제가 있다는 신호로 받아들여 리팩터링을 먼저 진행한다.
- **(If-Then)** 만약 도메인 로직 안에 숨은 의존(랜덤, 시간, 외부 호출 등)이 있다면 → 해당 의존을 바깥으로 꺼내 값으로 주입한 뒤 테스트한다.
- **(추가)** 1시간 이상 걸릴 시 '3. 테스트 어려움 대응' 보단 '4. 리팩터링 우선순위'를 준수한다.
- **(리팩터링 우선순위)**
    1) 역할 분리
    2) 숨은 의존성 제거
    3) 코드 가독성 향상 (변수명, 중복 제거, 상수화 등)

### 3. 책임 분리 및 객체의 책임 할당
- **(원칙)** 응집도와 결합도를 기준으로 설계를 판단한다. 객체는 단일 책임을 가져야 하며, 변경 이유 역시 해당 책임과 일치해야 한다.
- **(원칙)** "묻지 말고 시켜라(Tell, Don't Ask)" 원칙을 따른다. 즉, 행동은 해당 정보를 가장 잘 아는 객체가 수행한다.
- **(체크리스트)** 객체가 올바른 책임을 가졌는지 아래 기준으로 점검한다.
    - 클래스 상단의 `import` 문을 통해 책임 침범 여부를 점검한다.
    - 도메인 객체가 UI, 입출력, 프레임워크 의존성을 가지면 책임이 섞인 신호로 판단한다.
    - 일급 컬렉션이 담당해야 할 로직이 외부 서비스나 컨트롤러에 존재하지 않는지 확인한다.
    - 도메인은 항상 도메인 규칙과 상태 관리에만 집중하도록 유지한다.

### 4. 테스트와 설계 관련 규칙
- **(원칙)** 테스트하기 어려운 코드는 설계도 의심해봐야 한다.
- **(원칙)** 테스트는 구현이 아니라 **행위(Behavior)를 검증**해야 한다.
- **(금지)** 테스트를 위해 설계를 망치는 행위(getter 남발, 불필요한 public 메서드 노출 등)를 금한다.

### 5. 상태 위치 및 불변 객체 기준
- **(원칙)** 객체는 자신의 행동에 맞는 상태를 가져야 하며, 상태를 먼저 염두에 두고 행동을 설계하지 않는다.
- **(참조 불변)** `final` 키워드를 통해 참조 재할당을 방지하며, 컬렉션이나 배열의 의도치 않은 참조 변경을 막는 최소한의 안전 장치로 사용한다.
- **(내용 불변)** `setter`나 상태 변경 메서드를 제거하여 내부 데이터 변경을 차단한다. 기본 타입 또한 값 변경이 필요 없는 경우 `final`로 선언한다.

### 6. 캡슐화 및 보안 규칙
- **(원칙)** 객체에 접근할 때 퍼블릭 인터페이스를 기준으로 접근하며, 캡슐화를 위해 상태를 외부로 노출하는 행위를 금한다.
- **(금지)** `NullPointerException` 방지 및 검사 비용 절감을 위해 `null` 사용을 금한다.

### 7. 다형성 및 확장 규칙 (OCP)
- **(If-Then)** 만약 확장 시 기존 코드를 수정해야 하는 문제가 발생한다면 → 조건문을 다형성으로 대체하여 각 객체가 자신의 책임에 따라 행동하도록 위임한다.
- **(If-Then)** 만약 여러 동일한 책임이 있을 때 확장 가능한 설계를 원한다면 → 역할(인터페이스)을 도입한다.
- **(If-Then)** 만약 새로운 기물 타입을 추가한다면 → 다형성을 활용한 구체 클래스 추가와 보드 초기화 로직을 제외한 기존 도메인 코드는 수정하지 않는다.
